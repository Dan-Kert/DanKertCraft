# 🔍 Полный список оставшихся проблем DanKertCraft Launcher

## 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ

### 1. **Удаление инстанса БЕЗ подтверждения + race condition**
- **Файл**: `InstanceView.java` линия 312
- **Проблема**: 
  - `Files.walk().forEach(File::delete)` — нет подтверждения перед удалением
  - НЕТ асинхронности — удаление в UI потоке может зависнуть при большом размере
  - Возможна потеря данных если пользователь случайно нажал
- **Решение**: 
  - Добавить асинхронное удаление в фоновый поток
  - Добавить прогресс-бар удаления
  - Добавить возможность отмены

### 2. **Скачивание файлов БЕЗ ретраев при сетевых ошибках**
- **Файлы**: `FileDownloadHelper.java`, `GameInstaller.java`
- **Проблема**: 
  - При разрыве соединения файл скачивается неполностью
  - Нет повторных попыток при ошибке HTTP
  - Нет timeout'ов на скачивание больших файлов
  - Файл частично скачанный не удаляется
- **Решение**: 
  - Добавить exponential backoff retry (макс 3 попытки)
  - Удалять неполные файлы перед повтором
  - Поддержка resume (Range header) для больших файлов
  - Таймауты на соединение (сейчас только 10 сек)

### 3. **Утечка ресурсов в потоках**
- **Файлы**: `GameInstaller.java` (ExecutorService), `InstanceView.java`, `ModWindow.java`, множество новых Thread()
- **Проблема**: 
  - `ExecutorService` нигде не вызывается `.shutdown()` или `.awaitTermination()`
  - Множество `new Thread()` запускаются без имён (сложно отлаживать)
  - Потоки иногда не завершаются при исключении
  - Нет контроля кол-ва потоков в памяти
- **Решение**: 
  - Обернуть создание потоков в try-finally с гарантированным shutdown
  - Добавить имена потокам
  - Использовать ThreadFactory для логирования исключений
  - Ограничить кол-во одновременных потоков (ThreadPool вместо новых Thread)

### 4. **Пустые catch блоки и подавление ошибок**
- **Файлы**: `InstanceView.java` (320, 379), `GameLauncher.java` (407), `ModAPI.java` (33, 57), `OSHelper.java` (65)
- **Проблема**: 
  - `catch (Exception e) { e.printStackTrace(); }` — ошибка теряется
  - Нет логирования в `LogSystem`
  - Код пытается продолжить работу после ошибки без информации о причине
- **Решение**: 
  - Везде использовать `LogSystem.error("...", e)` 
  - Хотя бы минимально обработать ошибку или выбросить с контекстом

### 5. **Проблемы с File I/O и Windows путями**
- **Файлы**: `GameInstaller.java`, `FileDownloadHelper.java`
- **Проблема**: 
  - `File::delete` не логирует успех/неудачу удаления
  - На Windows файлы могут быть заблокированы процессом (Minecraft)
  - `mkdir()` используется вместо `mkdirs()` в некоторых местах
  - Нет проверки прав доступа перед операциями
- **Решение**: 
  - Всегда использовать `mkdirs()` для создания директорий
  - Проверять результат удаления и логировать
  - На Windows закрывать файловые хэндлы перед удалением
  - Попробовать шедулить удаление при перезагрузке если не удалось сразу

## 🟠 ВЫСОКИЙ ПРИОРИТЕТ

### 6. **Двойное создание GameInstaller**
- **Логи показывают**: Инициализация для платформы повторяется много раз
- **Причина**: `new GameInstaller(workDir)` вызывается в разных местах без кэширования
- **Решение**: Сделать singleton или хранить в UI контексте

### 7. **Отсутствие глобального обработчика исключений в потоках**
- **Файлы**: Везде где `new Thread`
- **Проблема**: 
  - Исключение в потоке не показывается пользователю
  - Молча падает фоновый процесс
- **Решение**: 
  ```java
  Thread thread = new Thread(() -> {...});
  thread.setUncaughtExceptionHandler((t, e) -> {
      LogSystem.error("Ошибка в потоке " + t.getName(), e);
  });
  ```

### 8. **Проверка целостности файлов БЕЗ кэша хешей**
- **Файлы**: `CacheManager.java`, `GameInstaller.java`
- **Проблема**: 
  - `calculateSHA1()` каждый раз пересчитывает хеш для БЕ́ЛЬШих файлов (GG, библиотеки)
  - Это медленно и блокирует UI
  - Нет кэша хешей между запусками
- **Решение**: 
  - Сохранять хеш в `cache/file_hashes.json` после загрузки
  - Проверять хеш только для новых файлов
  - Это может сэкономить минуты при запуске

### 9. **Синхронизация между LauncherUI и процессом игры**
- **Файлы**: `LauncherUI.java`, `GameLauncher.java`
- **Проблема**: 
  - Нет гарантии что процесс Java действительно запустился перед тем как скрыть launcher
  - `Process.pid()` может быть недостижимым на Linux
  - Нет проверки что процесс не завершился в первые 100мс
- **Решение**: 
  - Добавить небольшую задержку после запуска (100мс)
  - Проверить что `isAlive()` после запуска
  - Логировать PID если доступен

### 10. **Нет механизма для отмены загрузки**
- **Файлы**: `GameInstaller.java`, `InstallWindow.java`
- **Проблема**: 
  - `shouldStop` флаг есть, но нет способа его установить из UI
  - Пользователь не может отменить загрузку 60 ГБ ассетов
- **Решение**: 
  - Добавить кнопку "Отменить" в UI
  - Правильно завершать `ExecutorService` при отмене
  - Удалять неполные файлы после отмены

## 🟡 СРЕДНИЙ ПРИОРИТЕТ

### 11. **Нет валидации конфига при загрузке**
- **Файлы**: `ConfigManager.java`, `InstanceView.java`
- **Проблема**: 
  - Если конфиг испорчен JSON → crash
  - Нет default значений для важных полей
- **Решение**: 
  - Оборачивать парсинг JSON в try-catch
  - Возвращать default config если ошибка
  - Логировать ошибку для диагностики

### 12. **Множественные вызовы UI операций из разных потоков**
- **Файлы**: `LogWindow.java` (160 synchronized), `DownloadStatusBar.java`
- **Проблема**: 
  - `Platform.runLater()` нестабильно при частых вызовах
  - Может быть race condition в обновлении UI
- **Решение**: 
  - Использовать очередь сообщений вместо прямых вызовов
  - Батчить обновления (не более 10 в секунду)

### 13. **Нет таймаутов на сетевые операции**
- **Файлы**: `FileDownloadHelper.java`
- **Проблема**: 
  - Если сервер Mojang не отвечает → зависает навсегда
  - CONNECTION_TIMEOUT = 10 сек, но нет read timeout
- **Решение**: 
  - Добавить `connection.setReadTimeout(60000)`
  - Добавить общий таймаут на всю операцию

### 14. **Нет обработки прав доступа на Linux**
- **Файлы**: `GameLauncher.java`
- **Проблема**: 
  - `fixNativesPermissions()` использует `setExecutable(true, false)`
  - На Windows это игнорируется
  - На Linux может не хватить прав
- **Решение**: 
  - Проверить права перед запуском
  - Логировать если не удалось установить права
  - Предложить решение пользователю

### 15. **Нет поддержки 32-бит систем**
- **Файлы**: `JavaRuntimeManager.java`, `GameLauncher.java`
- **Проблема**: 
  - Код предполагает только 64-bit Java
  - На 32-бит машинах игра не запустится
- **Решение**: 
  - Проверить `System.getProperty("os.arch")`
  - Скачивать соответствующую версию Java
  - Логировать ограничение если 32-бит

## 🟢 НИЗКИЙ ПРИОРИТЕТ (Оптимизация)

### 16. **Отсутствие кэша для Meta информации**
- **Файлы**: `ModrinthAPI.java`, `GameInstaller.java`
- **Проблема**: 
  - Каждый раз запрашиваем список модов с API
  - Можно кэшировать на 1 час
- **Решение**: 
  - Добавить TTL кэш для API запросов

### 17. **Многократное парсирование JSON**
- **Файлы**: `GameInstaller.java`, `InstanceView.java`
- **Проблема**: 
  - Одна версия парсится несколько раз
  - `VersionData` каждый раз создаётся заново
- **Решение**: 
  - Кэшировать `VersionData` в памяти

### 18. **Неэффективная очередь логирования**
- **Файлы**: `LogSystem.java`
- **Проблема**: 
  - `LinkedBlockingQueue` с лимитом 1000 → может переполниться при большом кол-ве логов
  - Нет метрик переполнения очереди
- **Решение**: 
  - Использовать `ArrayBlockingQueue` с разумным размером
  - Логировать если очередь 80%+ заполнена
  - Может быть увеличить размер до 5000

### 19. **Нет прогресса при распаковке архивов**
- **Файлы**: `JavaInstaller.java` (вероятно)
- **Проблема**: 
  - Распаковка может занять минуты без прогресса
  - UI зависает
- **Решение**: 
  - Добавить прогресс-отчёты при распаковке

### 20. **Пути содержат пробелы → потенциальная проблема Windows**
- **Файлы**: Везде где используются пути
- **Проблема**: 
  - Хотя ProcessBuilder обрабатывает это, String конкатенация может сломать командную строку
- **Решение**: 
  - Всегда использовать `new File()` и `File.separator`
  - Избегать String конкатенации путей
